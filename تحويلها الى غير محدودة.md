

لأن المشكلة ليست فقط أن `Card` يحتاج variants. المشكلة أن اللغة تحتاج تفهم التصميم على خمس طبقات:

```text
Theme Identity
Component Variants
Layout Grammar
Section Personality
Custom Components + Scoped CSS
```

الأمثلة العشرة الحالية تثبت أن Amana قادرة تولّد صفحات مختلفة في الاتجاه والبنية والثيم والبيانات، مثل SaaS وeditorial وdeveloper console وmarketplace وdashboard وmedical booking وcreative studio. 
لكن التحليل البنيوي كشف أن اللغة حالياً ما زالت تعتمد كثيراً على مكونات مدمجة، ولا تدعم custom components، وهذه نقطة تقيد الحرية الحقيقية. 

الذي سأقوله لك الآن هو ما بعد الإصلاحات، كأنه **خطة بناء Amana v2 Design Engine**.

# أولاً: لا تبدأ من `Card(variant: glass)` مباشرة

هذا خطأ شائع.

لو بدأت تضيف variants مباشرة داخل `html.rs` هكذا:

```rust
if variant == "glass" { ... }
if variant == "brutal" { ... }
if variant == "luxury" { ... }
```

ستقع في نفس المشكلة بعد شهر: كود ضخم، مكرر، وصعب التوسيع.

الأفضل تبني نظام اسمه:

```text
Variant Registry
```

يعني كل مكون له سجل variants، وكل variant له tokens وstates وslots وresponsive behavior.

مثلاً بدل أن يكون `Card` hardcoded فقط، يصبح عندك داخلياً:

```rust
ComponentRegistry
  Card
    variants:
      minimal
      glass
      editorial
      brutal
      luxury
      neon
      dashboard
    states:
      default
      hover
      active
      selected
      disabled
    slots:
      media
      eyebrow
      title
      body
      actions
```

وقتها أي component جديد يستطيع استخدام نفس النظام.

# ثانياً: صمّم نظام Variant حقيقي

الصياغة التي عندك:

```amana
Card(variant: glass)
```

جيدة كبداية، لكن في لغة جادة تحتاج أكثر من هذا.

تحتاج أن تدعم:

```amana
Card(
    variant: glass,
    tone: violet,
    size: lg,
    density: airy,
    elevation: floating,
    radius: xl,
    hover: lift
)
```

لأن `variant` وحده لا يكفي. أحياناً تريد نفس `glass` لكن مرة violet ومرة emerald ومرة danger ومرة neutral.

الصياغة المتقدمة:

```amana
Card(
    variant: glass,
    tone: primary,
    surface: translucent,
    border: glow,
    shadow: deep,
    motion: lift
):
    h3: "إدارة الطلبات"
    p: "تابع كل طلب من لحظة الإنشاء حتى التسليم"
```

هذه تُترجم إلى HTML مثل:

```html
<article class="am-card am-card--glass am-tone-primary am-density-airy am-motion-lift">
  <h3>إدارة الطلبات</h3>
  <p>تابع كل طلب من لحظة الإنشاء حتى التسليم</p>
</article>
```

وتُترجم إلى CSS منظم:

```css
.am-card {
  border-radius: var(--radius-lg);
  padding: var(--space-lg);
  background: var(--surface-base);
  border: 1px solid var(--border-soft);
}

.am-card--glass {
  background: color-mix(in srgb, var(--surface-base) 72%, transparent);
  backdrop-filter: blur(22px);
  border-color: color-mix(in srgb, var(--color-primary) 28%, transparent);
}

.am-motion-lift {
  transition: var(--transition-smooth);
}

.am-motion-lift:hover {
  transform: translateY(-6px);
  box-shadow: var(--shadow-xl);
}
```

لاحظ أن هذا يحل مشكلة مهمة من التحليل: سابقاً `hover` لا يولّد `:hover` فعلياً، فلازم بعد الإصلاح يصبح عندك pseudo-rule compiler حقيقي.

# ثالثاً: أضف `variant` ككيان في اللغة، وليس فقط prop

المرحلة المتقدمة أن تسمح للمستخدم يعرّف variants بنفسه.

مثلاً:

```amana
variant Card.glass:
    base:
        surface: glass
        border: glow
        radius: xl
        shadow: soft

    hover:
        transform: lift
        shadow: large

    slots:
        title:
            font: display
            size: xl
        body:
            color: muted
```

و:

```amana
variant Card.brutal:
    base:
        surface: solid
        border: thick
        radius: sharp
        shadow: hard
        padding: lg

    hover:
        transform: offset
        border_color: accent
```

ثم يستخدمها المطور:

```amana
Card(variant: brutal):
    h3: "لوحة تحكم التجار"
    p: "كل شيء واضح، مباشر، بلا تعقيد"
```

هنا اللغة تصبح قابلة للتوسع. لا تحتاج أنت تضيف كل variant في Rust كل مرة.

داخلياً في AST:

```rust
pub struct VariantDecl {
    pub target: String,              // Card
    pub name: String,                // glass
    pub base: Vec<StyleDecl>,
    pub states: Vec<StateStyleDecl>, // hover, active, focus
    pub slots: Vec<SlotStyleDecl>,
    pub responsive: Vec<ResponsiveStyleDecl>,
}
```

وفي semantic validation:

```text
- هل Card موجود؟
- هل variant name صالح؟
- هل الخصائص معروفة؟
- هل hover يولد pseudo-class مسموح؟
- هل selector لا يهرب خارج scope؟
```

هذا يمنع العشوائية.

# رابعاً: ابنِ Layout Engine، لا تضف أسماء layouts فقط

وجود:

```amana
layout: bento
layout: masonry
layout: split
layout: asymmetric
```

جميل، لكن لو كل واحد مجرد class، ستبقى محدودة.

اللازم هو أن يكون عندك:

```text
Layout Primitive = algorithm + constraints + responsive behavior
```

مثلاً `bento` ليس مجرد grid. هو يحتاج:

```amana
section Features:
    layout: bento
    columns: 6
    rows: auto
    gap: lg
    responsive:
        desktop:
            columns: 6
        tablet:
            columns: 2
        mobile:
            columns: 1

    Card(span: 3x2, variant: glass):
        h3: "لوحة التجار"

    Card(span: 3x1, variant: minimal):
        h3: "إدارة المخزون"

    Card(span: 2x1, variant: neon):
        h3: "تنبيهات الطلبات"
```

هذا يعطي حرية حقيقية. ليس فقط “grid من 3 كروت”.

الكود المتولد:

```css
.section-features .am-layout-bento {
  display: grid;
  grid-template-columns: repeat(6, minmax(0, 1fr));
  gap: var(--space-lg);
}

.section-features [data-span="3x2"] {
  grid-column: span 3;
  grid-row: span 2;
}

@media (max-width: 768px) {
  .section-features .am-layout-bento {
    grid-template-columns: 1fr;
  }

  .section-features [data-span] {
    grid-column: auto;
    grid-row: auto;
  }
}
```

هكذا التصميم لا يتشابه، لأن نفس البيانات يمكن عرضها كبنتو، masonry، magazine، timeline، أو floating panels.

# سادساً: Section-level styling يجب أن يصبح نظام شخصية

الفكرة هنا ليست فقط:

```amana
section Pricing:
    mood: calm
```

بل يجب أن يكون `mood` يغيّر مجموعة قرارات تصميمية:

```text
mood: calm
  → low contrast
  → soft shadows
  → relaxed spacing
  → muted background
  → slow transitions

mood: energetic
  → stronger accent
  → sharper contrast
  → larger headings
  → active hover
  → gradient background

mood: editorial
  → readable width
  → serif/display heading
  → magazine columns
  → pull quotes

mood: luxury
  → dark surface
  → gold/neutral accent
  → thin borders
  → large whitespace

mood: playful
  → colorful surfaces
  → rounded cards
  → offset layout
```

الصياغة:

```amana
section Pricing:
    mood: calm
    layout: bento
    surface: glass
    emphasis: middle-card
    rhythm: balanced
    density: comfortable

    Card(variant: minimal):
        h3: "Starter"

    Card(variant: luxury, featured: true):
        h3: "Growth"

    Card(variant: minimal):
        h3: "Scale"
```

الكود المتولد يجب أن يكون scoped:

```css
.section-pricing {
  --section-surface: var(--surface-glass);
  --section-accent: var(--color-primary);
  --section-gap: var(--space-xl);
}

.section-pricing .am-card[data-featured="true"] {
  transform: translateY(-10px);
  border-color: var(--section-accent);
}
```

لا تجعل section styles عالمية. كل شيء لازم يكون scoped على:

```css
.section-pricing
```

أو:

```css
[data-section="pricing"]
```

# ثامناً: Safe custom CSS escape hatch يجب أن يكون Scoped DSL وليس CSS خام كامل

الصياغة التي كتبتها:

```amana
style Hero:
    .orb:
        position: absolute
        blur: 80
        opacity: 0.4
```

ممتازة، لكن لازم تضبطها بقواعد.

لا تسمح للمستخدم يكتب:

```amana
style Hero:
    body:
        display: none
```

ولا:

```amana
style Hero:
    *:
        all: unset
```

ولا:

```amana
style Hero:
    script:
        ...
```

الأفضل:

```amana
style Hero scoped:
    .orb:
        position: absolute
        inset_end: 10%
        top: 20%
        size: 320
        blur: 80
        opacity: 0.4
        background: accent

    .hero-card:hover:
        transform: lift
        shadow: large
```

ويتولد:

```css
.section-hero .orb {
  position: absolute;
  inset-inline-end: 10%;
  top: 20%;
  width: 320px;
  height: 320px;
  filter: blur(80px);
  opacity: 0.4;
  background: var(--color-accent);
}

.section-hero .hero-card:hover {
  transform: translateY(-6px);
  box-shadow: var(--shadow-large);
}
```

لاحظ أن selector صار تلقائياً:

```css
.section-hero .orb
```

وليس:

```css
.orb
```

هذه هي السلامة.

التحليل السابق ذكر أن `safeCssLiteral` كان صارماً زيادة ويرفض CSS صحيح مثل أسماء خطوط فيها apostrophe، لكنه أيضاً وضع فكرة فلترة الأشياء الخطيرة مثل `javascript:` و`expression()` و`<script`؛ هذا الاتجاه صحيح، لكن يجب دمجه مع scoping وproperty allowlist.

# تاسعاً: نظام أمان CSS لازم يكون 4 طبقات

لا تعتمد على regex واحد.

اعمل هكذا:

## 1. Selector safety

المسموح:

```text
.class
.class:hover
.class:focus-visible
.class .child
&:hover
& .child
```

الممنوع:

```text
body
html
*
#id
script
iframe
[onclick]
.class, body
```

## 2. Property allowlist

المسموح:

```text
display
position
inset
top
right
bottom
left
width
height
padding
margin
gap
grid
flex
background
color
border
radius
shadow
opacity
transform
filter
z_index
```

الممنوع أو المقيد:

```text
content
behavior
pointer-events
user-select
all
unicode-bidi
```

ليس بالضرورة تمنع كل شيء دائماً، لكن اجعل الخطر واضحاً.

## 3. Value sanitizer

امنع:

```text
javascript:
expression(
url(data:
url(http:
<script
</style
```

واسمح بـ:

```text
var(--token)
color-mix(...)
linear-gradient(...)
radial-gradient(...)
clamp(...)
calc(...)
```

## 4. Scope rewriter

أي selector يتحول تلقائياً إلى scope:

```text
.orb
```

يصير:

```css
.section-hero .orb
```

و:

```text
&:hover
```

يصير:

```css
.section-hero:hover
```

# عاشراً: Custom Components هي قلب الحرية الكاملة

هذه أهم مرحلة بعد variants/layouts.

لازم يستطيع المطور تعريف component:

```amana
component ProductCard(product: Product, variant: text = "market"):
    article.product-card:
        img(src: product.image, alt: product.name)
        div.content:
            h3: product.name
            p.price: product.price
            Badge(tone: success): product.status

    style:
        article:
            surface: card
            radius: xl
            padding: lg
            hover: lift

        .price:
            font: mono
            color: accent
```

ثم يستخدمه:

```amana
section Products:
    layout: masonry
    for product in products:
        ProductCard(product: product, variant: market)
```

هذا سينقل Amana من “لغة صفحات جاهزة” إلى “لغة بناء واجهات”.

داخلياً تحتاج:

```rust
pub struct ComponentDecl {
    pub name: String,
    pub params: Vec<ComponentParam>,
    pub body: Vec<ViewElement>,
    pub style: Option<ScopedStyleBlock>,
    pub variants: Vec<VariantDecl>,
}

pub struct ComponentParam {
    pub name: String,
    pub ty: Option<TypeRef>,
    pub default: Option<Expression>,
    pub required: bool,
}
```

وتحتاج component registry:

```rust
pub struct ComponentRegistry {
    pub builtins: HashMap<String, BuiltinComponent>,
    pub custom: HashMap<String, ComponentDecl>,
}
```

عند render:

```text
1. ابحث عن component بالاسم
2. إذا كان custom:
   - افحص props
   - اربط params بالقيم
   - ولّد scope class فريد
   - ولّد HTML من body
   - ولّد CSS scoped
3. إذا كان builtin:
   - استخدم renderer الداخلي
```

# حادي عشر: أضف slots للمكونات

بدون slots، custom components ستكون محدودة.

مثلاً:

```amana
component FeatureCard(title: str, icon: str):
    Card(variant: glass):
        Icon(name: icon)
        h3: title
        slot content
        div.actions:
            slot actions optional
```

الاستخدام:

```amana
FeatureCard(title: "إدارة الطلبات", icon: "package"):

        p: "تابع الطلبات من العميل إلى التوصيل"

    actions:
        Button(href: "/orders"): "عرض الطلبات"
```

هذا يسمح بمكونات مرنة جداً.

```rust
pub enum ViewElement {

        name: String,
        optional: bool,

        name: String,
        children: Vec<ViewElement>,
    },
}
```

```text
Component "FeatureCard" requires slot "content".
Slot "actions" is optional.
Unknown slot "footer". Available slots: content, actions.
```

```text
preset = theme + component defaults + layout defaults + motion defaults
```

# خامس عشر: أضف نظام “Design Intent” للذكاء الاصطناعي

بما أن Amana للفايب كودينج، لا تجعل المستخدم دائماً يحدد التفاصيل.

اسمح له يقول:

```amana
design:
    intent: "منصة تجار أردنية، فخمة، سهلة، تشبه تطبيقات التجارة الحديثة"
    audience: merchants
    mood: trustworthy
    complexity: medium
    uniqueness: high
```

الcompiler أو AI layer يترجم هذا إلى قرارات:

```text
- RTL
- dashboard shell
- strong order cards
- calm dark surface
- merchant-focused navigation
- clear status badges
- low visual noise
```

لكن مهم: هذا لا يكون compiler deterministic وحده. هذا جزء مناسب للـ AI planner قبل كتابة Amana.

يعني pipeline:

```text
User prompt
→ AI Design Planner
→ Amana DSL
→ Compiler
→ App
```

وليس:

```text
User prompt
→ React/CSS مباشر
```

هذه هي قيمة Amana الحقيقية.

# سادس عشر: أضف `design inspect` كأمر رسمي

بعد كل هذا، تحتاج أمر يفحص التشابه والمشاكل التصميمية:

```powershell
amana inspect examples/marketplace.amana --design
```

يعطي:

```json
{
  "ok": false,
  "score": 74,
  "issues": [
    {
      "level": "warning",
      "code": "REPEATED_SECTION_LAYOUT",
      "message": "4 sections use the same grid/card rhythm. Consider bento, split, or timeline."
    },
    {
      "level": "warning",
      "code": "LOW_VARIANT_DIVERSITY",
      "message": "Cards use only the default variant across the page."
    },
    {
      "level": "info",
      "code": "NO_SECTION_PERSONALITY",
      "message": "Section Pricing has no mood, emphasis, or surface."
    }
  ]
}
```

هذا مهم جداً. لا يكفي أن تدعم حرية التصميم. لازم اللغة تنبهك عندما التصميم صار متشابه.

معايير الفحص:

```text
- عدد layouts المختلفة في الصفحة
- عدد component variants المستخدمة
- تكرار نفس card rhythm
- وجود section mood
- وجود visual background مختلف
- توازن المسافات
- تباين الألوان
- responsive safety
- overflow risk
- empty states
- navigation clarity
```

# سابع عشر: أضف `style budget`

حتى لا يعطي الذكاء الاصطناعي تصميم مبالغ فيه.

مثلاً:

```amana
design:
    freedom: high
    style_budget:
        max_fonts: 2
        max_accent_colors: 3
        max_shadows: 4
        max_section_variants: 6
        motion: subtle
```

هذا يمنع الفوضى.

بدون style budget، الحرية الكاملة قد تتحول إلى صفحات مزعجة ومليانة effects.

# ثامن عشر: أضف Responsive Design كجزء من كل شيء

لا تجعل responsive فقط داخل `canvas`.

كل component وsection وlayout يجب أن يدعم responsive:

```amana
section Products:
    layout: sidebar
    responsive:
        desktop:
            layout: sidebar
            sidebar_width: 280
        tablet:
            layout: split
        mobile:
            layout: stacked
            filters: drawer
```

مثال قوي:

```amana
section Dashboard:
    layout: dashboard-shell
    responsive:
        desktop:
            sidebar: fixed
            metrics_columns: 4
        tablet:
            sidebar: collapsible
            metrics_columns: 2
        mobile:
            sidebar: drawer
            metrics_columns: 1
            bottom_nav: true
```

هنا Amana تصبح قادرة فعلاً على بناء تطبيقات، لا صفحات سطحية.

# تاسع عشر: أضف states لكل component

أي تصميم جاد يحتاج حالات:

```amana
Button:
    state loading:
        label: "جارٍ الحفظ..."
        icon: spinner
        disabled: true

    state success:
        label: "تم الحفظ"
        tone: success

    state error:
        label: "فشل الحفظ"
        tone: danger
```

أو:

```amana
Card:
    state selected:
        border: accent
        shadow: glow

    state disabled:
        opacity: 0.5
        pointer: none
```

وفي الاستخدام:

```amana
Button(state: loading): "حفظ"
```

أو ديناميكياً لاحقاً:

```amana
Button(state: form.submitting ? loading : default): "إرسال"
```

هذه المرحلة تدخل في interactivity، لكنها مهمة جداً لتصميم واقعي.

# عشرون: أضف Empty/Error/Loading كجزء من layout

لا تتركها للمطور كل مرة.

مثلاً:

```amana
section Orders:
    data: orders
    layout: table

    empty:
        title: "لا توجد طلبات بعد"
        description: "عندما يصل أول طلب سيظهر هنا"
        action: Button(href: "/products/new"): "أضف منتجاً"

    loading:
        skeleton: table rows 5

    error:
        title: "تعذر تحميل الطلبات"
        retry: true
```

هذا سيحل مشكلة كبيرة في المنتجات الحقيقية.

التحليل السابق ذكر غياب empty states وfeedback كجزء من ضعف التجربة، وهذا يجب أن يتحول إلى primitives في اللغة، لا مجرد CSS يدوي.

# واحد وعشرون: أضف Data-aware components

مثلاً بدل:

```amana
for product in products:
    Card:
        h3: product.name
```

اعمل:

```amana
ResourceGrid(products):
    item ProductCard(product)
    empty:
        title: "لا توجد منتجات"
    filters:
        - category
        - price
        - merchant
    sort:
        - newest
        - price_low
        - price_high
```

هذا يخدم مشاريع التجارة الإلكترونية جداً.

مثال لمنصة تجار:

```amana
view MerchantOrders:
    server:
        fetch orders = Order.where(merchant_id: currentUser.id)

    section Orders:
        layout: dashboard-shell

        ResourceTable(orders):
            columns:
                - order_number label "رقم الطلب"
                - customer_name label "العميل"
                - total label "القيمة" format money
                - status label "الحالة" as Badge
            actions:
                - view
                - update_status

            empty:
                title: "لا توجد طلبات حالياً"
                description: "ستظهر الطلبات الجديدة هنا فور وصولها"
```

هذا هو الفرق بين landing page DSL وproduct DSL.

# ثلاثة وعشرون: أضف Composition Rules

هذه قواعد تمنع التصميم المتشابه أو السيئ:

```text
Rule 1:
لا تستخدم layout grid في أكثر من 3 sections متتالية.

Rule 2:
إذا كان Hero split، اجعل القسم التالي إما bento أو editorial، لا split مرة ثانية.

Rule 3:
إذا الصفحة dashboard، لا تستخدم hero ضخم.

Rule 4:
إذا الصفحة medical/booking، الأولوية للثقة والوضوح، لا neon/glitch.

Rule 5:
إذا direction RTL، استخدم logical properties دائماً:
inset-inline-start
margin-inline
padding-inline
```

هذه يمكن أن تكون داخل `amana inspect`.

# أربعة وعشرون: أضف CSS Layers

بعد فصل CSS، استخدم CSS layers حتى لا تتقاتل الأنماط:

```css
@layer tokens, reset, base, components, variants, layouts, sections, pages, overrides;
```

وتوليد Amana يكون بهذا الترتيب:

```text
tokens.css       → @layer tokens
reset.css        → @layer reset
base.css         → @layer base
components.css   → @layer components
variants.css     → @layer variants
layouts.css      → @layer layouts
sections.css     → @layer sections
page.css         → @layer pages
custom style     → @layer overrides
```

هذا يحل مشاكل cascade.

والتحليل السابق أصلاً اقترح فصل CSS إلى `amana-tokens.css`, `amana-system.css`, وملفات pages مستقلة لأن CSS المدمج والمكرر غير قابل للصيانة.

# خمسة وعشرون: أضف Design Tokens حسب الصفحة

أحياناً تريد نفس التطبيق لكن صفحة مختلفة لها شخصية مختلفة:

```amana
view Landing:
    tokens:
        accent: violet
        surface: dark
        radius: xl

view Admin:
    tokens:
        accent: emerald
        surface: slate
        radius: md
        density: compact
```

لكن tokens هذه يجب أن تكون scoped:

```css
.view-landing {
  --color-accent: var(--violet-500);
}

.view-admin {
  --color-accent: var(--emerald-500);
}
```

وليس global.

# سبعة وعشرون: أضف Motion System

لا تجعل animation عشوائي.

```amana
motion:
    level: subtle
    duration: normal
    reduce_motion: respect
```

Component:

```amana
Card(motion: lift)
Modal(motion: scale)
Toast(motion: slide-up)
Section(motion: reveal)
```

CSS:

```css
@media (prefers-reduced-motion: reduce) {
  * {
    animation: none !important;
    transition-duration: 0.01ms !important;
  }
}
```

هذا ضروري للاحتراف.

# ثلاثون: أضف `escape hatch` على 3 مستويات

لا تجعل الخروج من النظام إما ممنوع أو مفتوح بالكامل.

## مستوى 1: Tokens override

```amana
tokens:
    accent: "#10b981"
    radius_lg: 28
```

## مستوى 2: Scoped style DSL

```amana
style Hero scoped:
    .orb:
        blur: 80
        opacity: .4
```

## مستوى 3: Raw CSS مقيد

```amana
css scoped Hero:
    .orb {
        filter: blur(80px);
        opacity: .4;
    }
```

لكن raw CSS يحتاج sanitizer وscope rewriter.

# ثمانية وثلاثون: الشكل النهائي الذي تريد الوصول له

تخيل ملف Amana بعد هذه المرحلة:

```amana
app SheinJordan:
    title: "Shein Jordan"
    db_path: "store.db"
    capabilities:
        - auth
        - api.rest

theme:
    mode: dark
    direction: rtl
    language: ar
    preset: commerce-dark
    accent: rose
    radius: soft
    density: comfortable

use amana/commerce
use amana/admin

model Product:
    name: str required
    price: money required
    image: str
    status: str default "active"

component ProductCard(product: Product, variant: text = "marketplace"):
    variants:
        marketplace:
            layout: vertical
            surface: glass
            image_ratio: square
            hover: lift

        compact:
            layout: row
            density: compact
            image_size: sm

    render:
        article.product-card:
            img(src: product.image, alt: product.name)
            h3: product.name
            Money(value: product.price)
            Badge(tone: success): product.status

    style scoped:
        .product-card:
            radius: xl
            overflow: hidden

view Home:
    design:
        archetype: marketplace-catalog
        mood: modern
        uniqueness: high

    section Hero:
        layout: split
        visual:
            background: radial-orbs
            decoration: floating-panels
        content:
            h1: "تسوق من محلاتنا في الأردن"
            p: "ابدأ من منتجاتنا الحالية، وقريباً من تجار أكثر"
            Button(tone: primary, href: "/products"): "تصفح المنتجات"

    section Products:
        layout: bento
        columns: 6
        responsive:
            mobile:
                layout: stacked

        for product in products:
            ProductCard(product: product, variant: marketplace)

        empty:
            title: "لا توجد منتجات حالياً"
            description: "سيتم إضافة المنتجات قريباً"
```

هذا المستوى يعطيك حرية حقيقية.

# أربعون: الخلاصة الصريحة

بعد حل المشاكل الحالية، لا تذهب مباشرة إلى “مزيد من components”.
اذهب إلى **نظام حرية تصميم كامل**.

الهدف ليس أن تقول:

```amana
Card(variant: glass)
```

فقط.

الهدف أن تصل إلى:

```amana
component + variant + layout + section mood + visual system + scoped style + responsive + inspect
```

عندها Amana تصبح قادرة على إنتاج:

```text
SaaS
Marketplace
Admin dashboard
Medical booking
Course platform
Event website
Agency portfolio
Developer console
E-commerce MVP
Merchant platform
```

بدون أن تشبه بعضها.

الجملة المهمة:

**إصلاح المشاكل يجعل Amana مستقرة.
Design Freedom Layer يجعل Amana حرة.
Custom Components يجعل Amana غير محدودة.**
