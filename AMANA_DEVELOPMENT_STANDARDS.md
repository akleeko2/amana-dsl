# الدليل الموسع لمعايير التطوير وهندسة واجهات المستخدم (AMANA_DEVELOPMENT_STANDARDS.md)

---

## مقدمة وميثاق جودة المشروع

يمثل هذا المستند القانون الأعلى والملزم لهندسة وتصميم واجهات وتطبيقات **Amana DSL**. يُلغي هذا الدليل أي محاولات للاجتهاد العشوائي، أو كتابة الكود السريع (Lazy Coding)، أو توليد واجهات بدائية تفتقر للثراء الجمالي والأداء الوظيفي المتكامل.

على أي مبرمج أو نموذج ذكاء اصطناعي (AI) يساهم في هذا المشروع أن يلتزم حرفياً بالمعايير الهندسية الواردة هنا قبل إجراء أي عملية دمج أو تجميع (Compilation). **مخالفة أي بند في هذا الدليل تعني فشل المراجعة البرمجية فوراً.**

## القسم 1: هندسة معمارية كود Amana DSL (Code Architecture)

لتجنب تحول المشاريع البرمجية الكبيرة إلى ملفات متضخمة يصعب تعديلها، يجب فصل الاهتمامات (Separation of Concerns) وفقاً للمخطط التالي:

```
apps/[app-name]/
├── app.amana               # ملف المدخل الرئيسي (إعلانات الاستيراد والتهيئة فقط)
├── config/
│   └── theme.amana         # توكينات التصميم والألوان والسمات المشتركة
├── models/
│   └── models.amana        # تعريف هياكل قواعد البيانات (Database Models)
├── seeds/
│   └── seeds.amana         # البيانات الأولية الواقعية لقاعدة البيانات
├── components/
│   └── components.amana    # المكونات الهيكلية المشتركة (Reusable Global Components)
├── routes/
│   └── routes.amana        # خريطة توجيه طلبات الصفحات والمتحكمات
└── views/
    ├── home.amana          # صفحة الهبوط العامة
    ├── dashboard.amana     # لوحة التحكم الرئيسية
    └── [other-views].amana # صفحات الأقسام الفردية
```

### أ) قواعد استيراد الملفات (Imports Hierarchy)
يجب أن يتم استيراد الملفات في `app.amana` بشكل منظم ومرتب منطقياً:
1. أولاً: ملفات الثيم والإعدادات (`config/theme.amana`).
2. ثانياً: ملفات النماذج وقواعد البيانات (`models/models.amana`).
3. ثالثاً: المكونات المشتركة (`components/components.amana`).
4. رابعاً: صفحات العرض المخصصة (`views/*.amana`).
5. خامساً: خريطة التوجيه (`routes/routes.amana`).

### ب) شروط بناء المكونات المشتركة (Component-Driven Standards)
* **قاعدة التكرار (Rule of Three):** إذا تم تكرار كتلة تصميمية (HTML/CSS) في أكثر من مكانين، **يجب** تحويلها فوراً إلى مكون مشترك داخل `components/components.amana` مع تمرير الخصائص كعوامل (parameters).
* **تخزين الحالة البصرية:** المكون المشترك لا يجب أن يتعامل مع الاستعلامات المباشرة لقاعدة البيانات (fetch)، بل يعتمد بالكامل على البيانات الممررة إليه من صفحة العرض (View) لضمان إعادة الاستخدام.

## القسم 2: الدليل البصري المتكامل وتوكينات التصميم (Design Tokens)

واجهات المستخدم الفاخرة لا تستخدم ألواناً عشوائية مطلقاً. تعتمد الواجهات الحديثة على فلسفة **تدرج الألوان المنسق بذكاء** والتأثيرات الزجاجية (Glassmorphic) لخلق تجربة ثلاثية الأبعاد جذابة.

### أ) لوحة الألوان الهيكلية وتناغم الخلفيات (Color Palette & Layering)

يجب الالتزام بالجدول التالي للألوان وتجنب استخدام أي كود لون ثابت (Hex color) خارج هذه القائمة:

| متغير التصميم (Token) | القيمة اللونية الافتراضية | التوصيف والاستخدام الوظيفي |
| :--- | :--- | :--- |
| `var(--bg-primary)` | `#0f172a` (Slate 900) | الخلفية الأساسية للتطبيق في الشاشات العادية وصفحات الهبوط. |
| `var(--bg-secondary)` | `#020617` (Slate 950) | خلفية التباين العميقة، وتستخدم لزيادة عمق لوحات التحكم والسايدبار. |
| `var(--color-surface-base)`| `#0f1117` (Dark Base) | الطبقة الأولى للواجهات والصفحات الإدارية. |
| `var(--color-surface-card)`| `#161b27` (Card Surface) | خلفية بطاقات المعلومات الأساسية، تظهر مرتفعة بصرياً عن سطح الصفحة. |
| `var(--color-surface-panel)`| `#1e2538` (Panel Surface) | أفتح درجات الواجهة، وتستخدم للوحات النبثقة (Modals) والعناصر العائمة. |
| `var(--color-brand)` | `#6366f1` (Indigo 500) | اللون الرئيسي للبراند والهوية، ويستخدم للإجراءات الأساسية (Primary CTA). |
| `var(--color-brand-soft)`| `#4f46e5` (Indigo 600) | النسخة الأكثر عمقاً من لون البراند، وتستخدم لحالات الـ Hover والتأكيد. |
| `var(--color-accent)` | `#22d3ee` (Cyan 400) | لون التمييز والإضاءة وجذب الانتباه (مثل مؤشرات النجاح والتأثيرات النيون). |
| `var(--color-border-subtle)`| `rgba(255, 255, 255, 0.07)`| الحدود الشفافة فائقة النعومة التي تفصل اللوحات والبطاقات دون إزعاج بصري. |
| `var(--color-success)` | `#10b981` (Emerald 500) | لون النجاح، العمليات المكتملة، المؤشرات الإيجابية، وحالات SLA الممتثلة. |
| `var(--color-warning)` | `#f59e0b` (Amber 500) | لون التحذير، المهام المعلقة، العمليات الجارية، وحالات ما قبل الاختراق. |
| `var(--color-danger)` | `#ef4444` (Red 500) | لون الخطأ الفادح، اختراق SLA، حذف البيانات، وحالات الطوارئ. |

### ب) التسلسل الهرمي للمشاهد (Visual Hierarchy & Elevation)
لكي يشعر المستخدم ببعد ومساحة العناصر، يجب تطبيق قيم الظلال والارتفاعات التالية:
* **الطبقة 0 (الخلفية العميقة):** لا تحتوي على ظلال، حوافها مدمجة بالكامل مع الأرضية.
* **الطبقة 1 (البطاقات واللوحات):** حوافها `1px solid var(--color-border-subtle)` مع ظل خفيف: `box-shadow: 0 4px 20px rgba(0, 0, 0, 0.3)`.
* **الطبقة 2 (العناصر العائمة والقوائم المنسدلة):** حوافها أفتح بدرجة طفيفة مع ظل أقوى: `box-shadow: 0 10px 32px rgba(0, 0, 0, 0.5)`.
* **الطبقة 3 (النوافذ المنبثقة - Modals):** تمنح خلفية ضبابية معتمة للمحتوى خلفها (Backdrop overlay) باستخدام: `backdrop-filter: blur(8px)`.

## القسم 3: الأبعاد الدقيقة وتخطيط الصفحة المتجاوب (Fluid Layouts & Responsiveness)

واجهات المستخدم البدائية تنكسر عند تصغير الشاشة أو تظهر أشرطة تمرير أفقية قبيحة. يجب تطبيق المعايير الرياضية التالية لضمان المرونة الكاملة:

### أ) محددات الخط البشري المريح وقاعدة الـ 72 حرفاً
* **حماية النصوص الطويلة:** لا تسمح للنصوص الطويلة أو الفقرات بالتمدد عبر الشاشة بالكامل، فهذا يجهد عين القارئ. يجب وضع حد أقصى لعرض مقالات المعرفة أو نصوص التذاكر:
  ```css
  .readable-text-container:
      max-width: "72ch"
      line-height: "1.75"
  ```
* **تفعيل كسر الكلمات الإجباري:** لمنع النصوص الطويلة والروابط من كسر أبعاد الحاويات، ضع دائماً هذه القواعد على العناوين والروابط:
  ```css
  .break-word-safe:
      overflow-wrap: "break-word"
      word-break: "break-word"
      max-width: "100%"
  ```

### ب) تقنية الخطوط المتدفقة بـ Math Clamp
يمنع استخدام قياسات الخطوط الثابتة (مثل `font-size: 24px` أو `18px`). يجب استخدام الصياغة الديناميكية التي تحسب حجم الخط بناءً على عرض جهاز العميل:
* **العناوين الرئيسية (H1):** `font-size: clamp(2rem, 5vw, 3.5rem)`
* **العناوين الفرعية (H2):** `font-size: clamp(1.5rem, 3.5vw, 2.2rem)`
* **عناوين اللوحات والبطاقات (H3):** `font-size: clamp(1.1rem, 2.2vw, 1.4rem)`
* **النص العادي (Body):** `font-size: clamp(0.9rem, 1.2vw, 1.05rem)`

### ج) معايير شبكة الـ Grid المتقدمة (Bento & Adaptive Grids)
عند إنشاء لوحة بيانات متجاوبة، يجب تحديد السلوك المتجاوب للشبكة لتجنب انضغاط الأعمدة:
* **صيغة الـ Grid المتجاوب التلقائي:**
  ```css
  .responsive-grid:
      display: "grid"
      grid-template-columns: "repeat(auto-fit, minmax(clamp(16rem, 25vw, 22rem), 1fr))"
      gap: "var(--space-lg)"
  ```
* **قاعدة stacked على الجوال:** يجب تهيئة التجاوب لتتحول العناصر إلى أسطر متكدسة فورياً عند انخفاض العرض عن حد معين:
  ```amana
  responsive:
      mobile: stacked
  ```

## القسم 4: معايير هيكلة شريط التمرير المقيد (Scroll-Lock Architecture)

**هذه هي القاعدة الأكثر أهمية لمنع تعطل التمرير واختفاء المحتوى السفلي في التطبيقات المعتمدة على السايدبار (Sidebar layout).**

### أ) البنية الكاملة للتحكم في التدفق الرأسي (Flexbox Overflow Boundary)
لكي يعمل التمرير الداخلي في منطقة العمل الرئيسية ولا تنقطع الأزرار أو ذيل الصفحة، يجب تطبيق التسلسل الهيكلي التالي في كود الأنماط (`style:`) لكل صفحة:

```
[Viewport 100dvh]
       │
       ▼ (overflow: hidden)
┌────────────────────────────────────────────────────────┐
│ .amana-runtime-shell                                   │
│  ├── div (x-data wrapper - overflow: hidden)           │
│  │    └── .app-shell (overflow: hidden)                │
│  │         ├── .side-rail (overflow-y: auto)           │
│  │         └── .dashboard-main (overflow-y: auto) <───┐│
│  │              │                                      ││
│  │              └─► [Scroll Content / Cards / Grid] ───┘│
└────────────────────────────────────────────────────────┘
```

ممثلاً بكود Amana DSL التالي (يجب نسخه وتطبيقه حرفياً على جميع صفحات العرض المعتمدة على لوحة التحكم):

```css
        .amana-runtime-shell:
            display: "block"
        
        # استهداف الحاوية المباشرة التي يولدها المترجم لـ Alpine.js
        .amana-runtime-shell > div:
            display: "flex"
            flex-direction: "column"
            height: "100%"
            min-height: "0"
            width: "100%"
            overflow: "hidden"
            
        # الهيكل الرئيسي للتطبيق ويشمل الشريط الجانبي ومحتوى العمل
        .app-shell:
            display: "flex"
            background: "#0b0d14"
            width: "100%"
            height: "100%"
            min-height: "0"
            overflow: "hidden"
            
        # منطقة المحتوى الحركي القابل للتمرير
        .dashboard-main:
            flex: "1"
            min-height: "0"
            overflow-y: "auto"
            display: "flex"
            flex-direction: "column"
            min-width: "0"
```

### ب) فئة الميزان المرن لبطاقات البيانات (Flex Grow & Shrink)
أي حاوية داخلية تحت `.dashboard-main` يجب أن لا تأخذ أطوالاً مطلقة تمنع التمرير، ويجب أن تحتوي دائماً على `min-height: 0` للسماح لمتصفح العميل باحتساب الارتفاع الديناميكي بدقة.

## القسم 5: الدليل الاحترافي للعناصر التفاعلية وتجربة العميل (UX/UI Components)

### أ) نموذج استمارة الإدخال الاحترافية (Professional Form Standards)
عند تصميم النماذج (Forms)، يمنع إدراج حقول مسطحة بدائية. يجب تطبيق الأنماط التالية:
1. **العناوين المساعدة (Labels):** يجب وضع تسمية واضحة لكل حقل بخط صغير ووزن عريض: `font-size: var(--text-xs); font-weight: 700; color: var(--color-ink-muted)`.
2. **حالة التركيز (Focus State):** عند قيام المستخدم بالنقر داخل الحقل، يجب أن يتغير لون الحدود وينشأ ظل متوهج ناعم (Glow Ring) يوضح مكان المؤشر:
   ```css
   .form-input:focus:
       border-color: "var(--color-brand)"
       box-shadow: "0 0 0 3px rgba(99, 102, 241, 0.15)"
       outline: "none"
   ```
3. **حالة عدم التفعيل (Disabled State):** إذا كان الحقل مغلقاً، يجب تغيير الشفافية ومؤشر الفأرة: `opacity: 0.5; cursor: not-allowed`.

### ب) قواعد تنسيق الجداول والبيانات (ResourceTable Standards)
الجداول هي العمود الفقري لأي نظام تذاكر أو بيانات عملاء. لضمان سلاسة قراءتها:
* **التمرير الأفقي للجداول المزدحمة:** يجب تغليف أي جدول داخل حاوية مخصصة تسمح بالتمرير الأفقي على الشاشات الصغيرة فقط لمنع تشويه الصفحة:
  ```css
  .table-responsive-wrapper:
      width: "100%"
      overflow-x: "auto"
      -webkit-overflow-scrolling: "touch"
  ```
* **تناوب ألوان الصفوف (Zebra Striping):** يسهل تتبع الأسطر البصرية بالتناوب الخفيف:
  `background: rgba(255, 255, 255, 0.01)` والسطر الآخر `background: rgba(255, 255, 255, 0.03)`.
* **تباين مؤشر الأولوية (Priority & Status Badges):** لا تستخدم نصوصاً مجردة للأولويات. استخدم دائماً مكون البادج الملون المناسب مع شفافية خلفية ناعمة:
  - أولوية قصوى (`urgent`/`high`): بادج أحمر داكن مع كتابة حمراء زاهية.
  - أولوية عادية (`medium`): بادج أمبر/برتقالي مع كتابة ذهبية.
  - أولوية منخفضة (`low`): بادج رمادي أو أزرق خفيف.

### ج) الأزرار والتفاعلات الدقيقة (Interactive Elements)
* **المسافة الآمنة للنقر (Touch Targets):** الحد الأدنى لعرض وارتفاع أي زر تفاعلي أو رابط يجب ألا يقل عن `44px` لضمان سهولة النقر بأصابع اليد على شاشات اللمس.
* **الانتقال السلس (Transitions):** أي تغيير في اللون، الحجم، أو الحدود لعنصر تفاعلي يجب أن يستغرق ما بين `120ms` إلى `200ms` كفترة انتقالية لتجنب التغيرات الفجائية المشوهة بصرياً:
  ```css
  .interactive-element:
      transition: "all 150ms ease-in-out"
  ```

## القسم 6: قواعد سلامة البيانات وجودة التهيئة (Database & Seeds Realism)

### أ) واقعية البيانات الأولية (Zero Lorem Ipsum Rule)
يمنع استخدام أي كلمات وهمية مثل "Lorem Ipsum" أو أحرف عشوائية في قاعدة البيانات أو التذاكر الأولية. يجب كتابة بيانات واقعية تحاكي سيناريوهات العمل الحقيقي:
* **أسماء العملاء:** أسماء ثلاثية عربية أو أجنبية متناسقة.
* **مواضيع التذاكر:** عناوين حقيقية لشكاوى تقنية (مثال: "تعطل بوابة الدفع لبطاقات مدى"، "مشكلة في تحميل فاتورة الاشتراك الشهري").
* **البريد الإلكتروني:** يجب أن يكون بريداً رسمياً ذو شكل واقعي (مثال: `customer.support@company.com`).

### ب) سلامة المعرفات والربط
* **حقول التحديث الإجبارية:** في كل نموذج لتعديل البيانات (Update Forms)، يجب وضع عنصر إدخال مخفي (hidden input) يحتوي على المعرف الرئيسي للسجل (`id`) لضمان عدم حدوث أخطاء خادم عند معالجة طلبات الـ POST:
  ```amana
  form [name, status, id]:
      connect Agent.update
      where id = Agent.current.id
      default id = Agent.current.id
  ```

## القسم 7: دليل استكشاف الأخطاء للمترجم (Compiler Troubleshooting & Diagnostic Guide)

في حال واجهتك مشاكل أثناء تجميع أو تشغيل الكود، يوضح هذا الجدول الخطأ والتشخيص والحل الفوري:

| الخطأ البنائي للمترجم | التشخيص التقني للخلل | الإجراء العلاجي الفوري |
| :--- | :--- | :--- |
| `Unexpected character '﻿' (0xfeff)` | وجود علامة ترتيب البايتات (BOM) في بداية الملف بسبب حفظه من محرر ويندوز أو PowerShell بشكل خاطئ. | أعد كتابة وحفظ الملف بصيغة UTF-8 صافية (بدون BOM) باستخدام كود بايثون مخصص أو محرر يدعم الحفظ النظيف. |
| `Access is denied. (os error 5)` | محاولة تشغيل أمر الفحص أو البناء وتوجيهه إلى مجلد أو مسار خاطئ بدلاً من توجيهه لملف البداية `app.amana`. | قم بتشغيل الأمر موجهاً بدقة للملف الرئيسي: `cargo run -- build apps/[app-name]/app.amana ...` |
| `Parser Error: Unexpected token` | وضع نقطتين رأسيتين `:` بعد مكون مغلق ذاتياً، أو استخدام صياغة غير مدعومة مثل `mobile.columns` داخل responsive. | قم بمراجعة علامات الترقيم بعد المكونات، وتعديل صيغ التجاوب لتعمل بأسلوب التكديس المدعوم `mobile: stacked`. |
| `Redirect syntax failure` | استخدام متغيرات ديناميكية أو f-strings في مسار التوجيه للنجاح أو الفشل. | استبدل المسار الديناميكي بمسار ثابت مستقر مثل: `redirect success -> /dashboard`. |

## ميثاق الالتزام النهائي

تذكر دائماً: **الواجهات البسيطة والبدائية التي تبدو كمسودة هي دليل على الفشل التام في تبني هذا المشروع.**
احرص دائماً على تطبيق:
1. جمالية الهياكل الزجاجية (Glassmorphic Panels).
2. الألوان المستمدة من المتغيرات فقط (Variables).
3. تقييد التمرير وحماية أبعاد الهيكل (Scroll Boundary).
4. التجاوب الكامل والمحاذاة الذكية للعناصر في الشاشات المختلفة.
